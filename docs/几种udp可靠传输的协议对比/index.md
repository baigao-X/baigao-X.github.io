# KCP

纯算法协议，不负责底层的socket收发。
## 目标

以额外的带宽，来获得更低延时的传输体验。适用与单路对带宽的充分压榨，但是不利于全局的带宽分配使用。

## 机制
- TCP采用的是UNA(表示指定序号之前的包都收到了)，因此丢包会重传丢包后的所有数据，KCP采用ACK(表示指定序号的包收到了)+UNA, 因此选择性重传，只重传丢的包，减少冗余
- 快速重传(根据ACK来知道丢失的包，不等待超时直接重传)
- 动态RTO调整 （默认TCP超时是按照RTC x 2来设置），KCP采用x1.5。 同时最小RTO设置为10ms（TCP默认是100ms）
- TCP会为了利用带宽 延迟发送ACK(合并发送ACK，以减少ACK数据包)



# SRT

## 目标
可靠低延迟的流媒体传输

## 机制
- NAK(否确认)机制： 根据rtt，来定时发送NAK包，NAK包指定丢失的包要求重传。
- 通过ACK和ACKACK来计算rtt。
- AES加密， 定时更新密钥
- FEC前向纠错

# QUIC

## 目标
替代TCP

## 机制
- TLS加密
- 多路复用
- 0-RTT 握手（原来TCP握手需要1个RTT，TLS1.2需要2个，TLS1.3需要1个，因此https需要2-3个）。
	- QUIC通过合并TCP握手和TLS握手，首次连接仅需要1个RTT即可。
	- 核型在于客户端缓存前向安全密钥材料和传输参数。因此同一客户端连接同一服务器，可以直接连接无需重新协商。
- FEC 前向纠错

# WEBRTC

## 目标
实时音视频通道

- NAT穿透
- 流控依赖RTCP的 NACK 和PLI等机制实现选择性重传
