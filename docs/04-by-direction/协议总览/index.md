# 1. 基于FLV封装的
## 1.1. RTMP 

基于TCP
基于FLV封装，一般用于直播源推流，直播内流数据传递
既可以推流，也可以拉流
rtmp://
因为浏览器摈弃了Flash播放器，所以没法做浏览器观看
封装音视频数据强制切片，每个包不会太大，失败重发成本不大，有一定弱网抵抗能力，但是服务端需要合并切片，会加大CPU压力
低延迟(1-3s)
## 1.2. HTTP-FLV

基于FLV封装，一般只用于客户端直播流观看
可以简单理解为RTMP的HTTP版本，也切片,内存发送，不需要切片成静态文件
低延迟(1-3s)，比RTMP协议略高
http://
一般需要服务端加入flv插件才能观看

# 2. HLS 协议

http://  ....m3u8

一般只做拉流观看，可用于直播、点播
本质上是通过HTTP协议下载静态文件 （m3u8索引文件以及一系列几秒一个的ts切片文件）
加入HLS的js插件即可，苹果设备原生支持hls
因为hls文件是无状态的静态文件，且大小有限，所以负载均衡、CDN加速效果明显，点播上比直接点播MP4、flv
更快播放出来，且跳转更顺滑
直播下没什么优势，延时在5-30秒
直播时移(直播转点播)时有优势

# 3. WEBRTC 
主打低延时
UDP通道，低延时，且自带丢包、乱序、延时等机制。 
延迟比RTMP低，大概1秒内
并不是为了直播场景设计的，是为了点对点视频/语音通话协议
交互式低延时直播协议，需要用到WebRtc
支持推流和拉流
成本高
webrtc://

# 4. RTSP

一般不用做直播场景，现代浏览器不能播，web领域一般不用rtsp
一般用作摄像头等硬件设备的实时流观看、推送
有TCP/UDP切换，支持直播点播等优点
命令、流分离，需要多个通道
一般延时比RTMP低




# 5. SRT

可以点对点传输
安全稳定
能适应复杂环境，抗干扰
低延时，带宽利用率高
适用于低带宽、高丢包环境下保证传输质量和稳定性，提高稳定性和流畅度
