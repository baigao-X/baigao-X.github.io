## 表设计

### 属性
#### 候选码
能决定唯一一行记录的属性组

#### 主键 (主属性)
**primary key**
从候选码中挑选出来，主键可以是一列或者多列组成的，只要能够根据主键，马上能精确到特定的一行数据即可。
主键内存之不能为空，一个表只能有一个主键
选取主键的一个基本原则是：**不使用任何业务相关的字段作为主键**

#### 外键
**foreign key**
外键约束会降低数据库的性能，大部分互联网应用程序为了追求速度，并不设置外键约束，而是仅靠应用程序自身来保证逻辑的正确性

### 关系
###### 函数依赖

如果在一张表中，只要X属性的值确定，那么就能确定Y属性，那么称为Y依赖于X，记作**X→Y**
- **部分函数依赖**
	- 指一个非主属性依赖于**符合主键**的一部分，而不是整个符合主键。
	- 假设一个学生成绩表，其中主键由 "学号" 和"课程ID"两个字段组成，如果”学生姓名“只依赖于“学号”而不依赖于“课程ID”，就是部分依赖
	- 记作**X F→ Y**: 如 （学号，课程ID） P→ 学生姓名
- **完全函数依赖**
	- 完全依赖是指一个非主属性依赖于整个复合主键，而不是仅依赖于主键的一部分。
	- 假设一个学生成绩表，其中主键由 "学号" 和"课程ID"两个字段组成，如果”成绩“依赖于“学号”和依赖于“课程ID”，就是完全依赖
	- 记作**X F→ Y**: 如 （学号，课程ID） F→ 成绩
- **传递函数依赖**：
	- 假如 Z 函数依赖于 Y，且 Y 函数依赖于 X ，那么我们就称 Z 传递函数依赖于 X 
	- 记作 **X T→ Z**

### 设计原则
#### 范式

遵循范式设计的表有以下优点：
 - 减少数据冗余
 - 数据表更新快，体积小


##### 第一范式（1NF）
- 每一列属性都是不可再分的属性值，两列的属性相近或相似或一样，尽量合并属性成一样的列
- 保证数据的**原子性**
###### 举例:
- 不满足第一范式：
	**姓名**: 张三,  **地址**:江苏省南京市浦口区xxx街道xxx小区,**省**: 江苏省,**市**:南京市
- 修改为符合第一范式:
	**姓名**: 张三,  **地址**:xxx街道xxx小区,**省**: 江苏省,**市**:南京市,**区**:浦口区
##### 第二范式（2NF）

- 满足第一范式
- 消除**部分依赖**问题,保证数据的**唯一性**
###### 优点：
 - 减少数据冗余
 - 避免数据异常
 - 提高数据一致性

###### 步骤
- 识别复合主键
- 识别部分依赖
- 拆分表结构
- 建立**外键关系**
###### 举例:
- 不满足第二范式：
	**学号, 学生姓名,课程ID,课程名称，成绩**
- 修改为符合第二范式:
	学号, 学生姓名**
	**课程ID,课程名称**
	**学号, 课程ID,成绩**
##### 第三范式（3NF）:属性完全依赖于主键
- 满足第二范式
- 消除**传递依赖**问题,保证数据的**非冗余性**。 即不能存在非主属性依赖于非主属性的情况
###### 举例:
- 不满足第三范式：
	**学号, 学生姓名,年龄,学院名称,学院电话**
- 修改为符合第三范式:
	**学号, 学生姓名,年龄,所在学院**
	**学院,学院名称,学院电话**

## 高级用法

### 约束
约束是在表的数据列上强制执行的规则，这些是用来限制可以插入到表中的数据类型，这确保了数据库中数据的准确性和可靠性。
- **NOT NULL 约束**：确保某列不能有 NULL 值。
- **DEFAULT 约束**：当某列没有指定值时，为该列提供默认值。
- **UNIQUE 约束**：确保某列中的所有值是不同的。
- **PRIMARY Key 约束**：唯一标识数据库表中的各行/记录。
- **CHECK 约束**：CHECK 约束确保某列中的所有值满足一定条件。

### 连接 join

##### 交叉连接 - CROSS JOIN
对两个表做**笛卡尔积**操作

##### 内连接 - INNER JOIN
##### 外连接 - OUTER JOIN

### 触发器（Trigger）
SQLite 触发器是数据库的回调函数，它会在指定的数据库事件发生时自动执行/调用。
### 事务（Transaction）
- 事务处理可以用来维护数据库的完整性，保证成批的 SQL 语句要么全部执行，要么全部不执行。

### 索引(Index）
索引（Index）是一种特殊的查找表，数据库搜索引擎用来加快数据检索。
**隐式索引**是在创建对象时，由数据库服务器自动创建的索引。索引自动创建为主键约束和唯一约束。

#### 什么情况下要避免使用索引？

虽然索引的目的在于提高数据库的性能，但这里有几个情况需要避免使用索引。使用索引时，应重新考虑下列准则：
- 索引不应该使用在较小的表上。
- 索引不应该使用在有频繁的大批量的更新或插入操作的表上。
- 索引不应该使用在含有大量的 NULL 值的列上。
- 索引不应该使用在频繁操作的列上。
### 视图(View)
视图（View）只不过是通过相关的名称存储在数据库中的一个 SQLite 语句。视图（View）实际上是一个以预定义的 SQLite 查询形式存在的表的组合。
